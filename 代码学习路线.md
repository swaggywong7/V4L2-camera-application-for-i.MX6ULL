## 阶段一：先理解数据流（30分钟）

在读任何代码之前，先在脑子里建立这条数据链路：

```
摄像头硬件
    ↓ (驱动把数据写入内核缓冲区)
V4L2Device::dequeue_buffer()  →  拿到MJPEG原始字节流
    ↓
CaptureThread::decode_mjpeg()  →  用cv::imdecode解码为cv::Mat
    ↓
回调到CameraApp::on_frame_captured()
    ↓ (判断是否需要处理)
    ├─ 不处理 → 直接送FrameBuffer::display()
    └─ 需要处理 → ImageProcessor::submit_frame()
                        ↓ (condition_variable唤醒处理线程)
                  ProcessAlgorithm::process()  →  算法处理
                        ↓
                  回调到CameraApp::on_frame_processed()
                        ↓
                  FrameBuffer::display()  →  写入显存 → 屏幕显示
```

**你要能在白板上画出这张图**，并且解释每个箭头处发生了什么。





 三者关系

  主线程（老板）
    ├── 创建/销毁 采集线程和处理线程
    ├── 接收用户命令
    └── 管理所有组件生命周期

  采集线程（搬运工）
    ├── 从摄像头取帧
    ├── 回调给主线程的CameraApp对象（但注意：回调代码跑在采集线程上）
    └── 如果需要处理，把帧扔给处理线程

  处理线程（加工工）
    ├── 等采集线程送帧
    ├── 做图像处理算法
    └── 回调把结果送到framebuffer显示

  数据流向： 摄像头 → 采集线程 → 处理线程 → framebuffer 显示

## 阶段二：从底层往上读（按这个顺序）

### 第1步：V4L2Device（最核心，必须吃透）

**读：** `include/v4l2_device.h` → `src/v4l2_device.cpp`

**设计时的思考过程：**

> "我需要操作V4L2摄像头。原来的代码把open、ioctl、mmap、close散落在v4l2类的各个成员函数和析构函数里，靠一个`start`标志位判断资源是否需要释放。我觉得这样很脆弱——万一中间某步出错跳出去了，后面的释放就跳过了。
>
> 所以我的第一个想法是**把设备操作封装成一个类，用析构函数保证释放**。这就是RAII。
>
> 然后我想到：一个摄像头设备应该只能被一个对象持有，不能拷贝。两个对象持有同一个fd，一个析构了另一个就操作野指针了。所以我**删除了拷贝构造和拷贝赋值**。但是可能需要转移所有权（比如从工厂函数返回），所以**保留了移动语义**。移动的时候把原对象的fd置为-1，这样原对象析构时close(-1)不会有副作用。
>
> Frame结构体的设计也是有考虑的——原来用裸指针+长度+index三个散落的变量传来传去，容易搞混。我把它们打包成一个结构体，语义更清晰。"

**重点理解：**
- `open()` 里的分步初始化：每步失败都要清理前面已获得的资源
- `request_buffers()` 里先QUERYBUF再mmap再QBUF的顺序
- 析构函数里 close → stop_streaming → release_buffers → ::close 的顺序（**反序释放**）

**自检问题：**
1. 为什么mmap时用MAP_SHARED而不是MAP_PRIVATE？
   → 因为需要和驱动共享这块内存，驱动写入数据后用户空间能看到
2. VIDIOC_QBUF和VIDIOC_DQBUF的Q和DQ分别代表什么？
   → Queue（入队）和Dequeue（出队），是内核缓冲区队列的操作
3. 为什么析构函数里要先stop_streaming再munmap？
   → 如果还在采集中，驱动可能正在往映射内存写数据，先停流才安全

### 第2步：CaptureThread（理解线程模型）

**读：** `include/capture_thread.h` → `src/capture_thread.cpp`

**设计时的思考过程：**

> "采集需要在独立线程中进行，因为DQBUF是阻塞调用——如果放在主线程，整个程序就卡住了。
>
> 原来用的QThread，现在去掉Qt了，自然换成std::thread。但线程间怎么通信？原来用Qt信号槽，现在我用**std::function回调**。本质上信号槽就是一种回调机制，只是Qt帮你处理了跨线程的消息队列。
>
> 回调函数用std::function包装，这样调用者可以传lambda、函数指针、成员函数绑定——比纯函数指针灵活。注册回调时要加锁保护，因为主线程注册、工作线程调用，是两个线程。
>
> 停止线程用atomic<bool>——我特意没有用普通bool。面试的时候说原因：ARM内存模型比x86弱，普通bool在ARM上可能看不到另一个核写入的值。
>
> MJPEG解码我直接用OpenCV的imdecode，一行代码。原来的代码先用QPixmap::loadFromData解码成QPixmap，然后在处理线程里又转成cv::Mat，中间经过了QImage——做了3次格式转换。我直接从原始数据解码到cv::Mat，省掉了所有中间转换。"

**重点理解：**
- `capture_loop()` 里的循环：DQBUF → 解码 → 回调 → QBUF，注意QBUF放在回调之后（**无论解码是否成功都要归还缓冲区**）
- `stop()` 里先设标志再join的模式
- `decode_mjpeg()` 用 `cv::imdecode(buf, IMREAD_COLOR)` 一步到位

**自检问题：**
1. 为什么CaptureThread持有V4L2Device的引用而不是指针？
   → 引用意味着"一定存在且不为空"，语义更强。指针可能为nullptr需要额外检查
2. 如果摄像头被拔掉了会怎样？
   → DQBUF返回-1，errno不是EINTR，进入error分支，通过error_callback通知上层

### 第3步：ImageProcessor（策略模式 + 生产者消费者）

**读顺序：** 先读 `image_processor.h` 里的 `ProcessAlgorithm` 基类和各派生类声明 → 再读 `ImageProcessor` 类声明 → 最后读 `image_processor.cpp` 实现

**设计时的思考过程：**

> "我有7种图像处理算法，原来全写在一个switch-case里。问题是：每次加新算法就要改这个switch，而且所有算法的代码挤在一个文件里，很长。
>
> 我想到用**策略模式**——定义一个算法接口ProcessAlgorithm，每种算法继承它实现自己的process()。然后用一个map把模式枚举映射到算法对象。运行时查表调用就行了，不需要switch。
>
> 为什么用unordered_map而不是数组？因为枚举值不一定连续，而且map的语义更清晰——它表达的是"模式→算法"的映射关系。性能差异在这里完全不重要，查一次map是O(1)，图像处理本身要几十毫秒。
>
> 算法对象用unique_ptr持有，因为map拥有这些算法，算法不需要被共享。
>
> 处理线程的设计是**生产者-消费者**。原来用msleep(10)轮询，即没有帧也每10ms醒一次，浪费CPU。我改用condition_variable：采集线程submit_frame时notify_one，处理线程wait等待。没有帧就真的挂起，CPU占用为零。
>
> 有个细节：我用wait_for而不是wait。如果只用wait，当采集停止后不再有notify，处理线程就永远卡在wait上了，stop()的join()就会死锁。wait_for最多等100ms就会返回，这时候检查running_标志，如果false就退出循环。"

**重点理解：**
- `register_algorithms()` 的注册表模式：扩展只需加一行
- `submit_frame()` 的 notify_one 和 `process_loop()` 的 wait_for 配合
- wait_for 的第三个参数（谓词lambda）：防止虚假唤醒（spurious wakeup）

**自检问题：**
1. 什么是虚假唤醒？为什么wait需要配合谓词？
   → POSIX标准允许condition_variable无理由返回，谓词确保真正有数据才继续
2. 为什么submit_frame里要clone()？
   → 采集线程的frame可能在下一帧到来时被覆盖，clone确保处理线程拿到的是独立副本
3. current_mode_为什么用atomic而不是加锁保护？
   → 它只是一个枚举值的读写，atomic比mutex更轻量。读和写是不同线程，但都是简单的load/store

### 第4步：FrameBuffer（Linux显示机制）

**读：** `include/frame_buffer.h` → `src/frame_buffer.cpp`

**设计时的思考过程：**

> "去掉Qt之后，我需要一种方式把图像显示到IMX6ULL的屏幕上。Linux提供了两种方式：FrameBuffer和DRM/KMS。FrameBuffer更简单，虽然是旧API但在嵌入式场景下完全够用。
>
> FrameBuffer的本质就是把显存映射到用户空间，写内存就等于写屏幕。关键是要弄清楚像素在内存中的布局：`offset = y * line_length + x * (bpp/8)`。注意line_length不等于xres * bpp/8——硬件可能有行对齐（比如按64字节对齐），所以要用ioctl获取的实际line_length。
>
> 色深方面要处理两种常见格式：ARGB8888是每像素4字节，直接写BGRA就行；RGB565是每像素2字节，需要把RGB888压缩为5-6-5位。IMX6ULL通常用ARGB8888。
>
> display函数里我先用cv::resize缩放图像保持比例，然后居中显示。最初用的是逐像素写入，后来做了性能优化：ARGB8888模式下先用cv::cvtColor一次性把BGR转为BGRA，再按行memcpy写入显存——从30万次函数调用减少到480次（640x480分辨率）。RGB565模式下按行遍历做位移压缩。"

**自检问题：**
1. line_length和xres*bpp/8有什么区别？
   → 硬件可能对每行做对齐填充，line_length是实际每行字节数，可能比xres*bpp/8大
2. 为什么写完像素画面就变了，不需要"刷新"？
   → mmap是共享映射，写入立刻反映到显存，显示控制器下次刷新时就会读到新数据

---

### 第5步：PhotoManager（文件操作）

**读：** `include/photo_manager.h` → `src/photo_manager.cpp`

**设计时的思考过程：**

> "拍照功能原来散落在v4l2类里，相册浏览在showphoto类里，同样的目录操作代码写了两遍。我统一封装成PhotoManager。
>
> 用C++17的std::filesystem替代Qt的QDir。文件名生成用时间戳+毫秒数，比原来的10位随机数更有意义——从文件名就能看出拍照时间，而且碰撞概率更低（同一毫秒内拍两张照片几乎不可能）。
>
> save_raw直接写原始MJPEG数据（本身就是JPEG格式），save_mat用cv::imwrite编码保存——两种接口覆盖不同场景。"

这个类最简单，快速过一遍即可。

---

### 第6步：CameraApp（总控制器）

**读：** `include/camera_app.h` → `src/camera_app.cpp`

**设计时的思考过程：**

> "我需要一个东西把所有组件串起来。原来是通过全局变量和Qt信号槽隐式连接的，耦合度很高。
>
> CameraApp就是协调者（Mediator角色）。它用unique_ptr持有所有组件，在handle_open()里按正确顺序创建和启动它们，在handle_close()里按反序停止和释放。
>
> 回调的注册使用lambda捕获this指针，比如`[this](const cv::Mat& frame) { on_frame_captured(frame); }`。这本质上就是Qt connect的手动版本。
>
> on_frame_captured里有一个分流逻辑：如果处理模式是None，直接显示；否则提交给处理线程。这就是原来Qt槽函数updateFrame()的逻辑，但不再依赖信号槽机制了。"

**重点理解：**
- `on_frame_captured()` 里先 clone 缓存帧（给拍照用），再分流到处理/显示
- `handle_capture()` 从 `cached_frame_` 取帧，**不直接调 dequeue_buffer**——避免与采集线程竞争同一个V4L2设备
- `run()` 里对 `std::stoi` 加了 try-catch，防止用户输入非数字时崩溃
- `request_stop()` 是 async-signal-safe 的（只做一次 atomic store），信号处理器可以安全调用

**自检问题：**
1. 为什么display_mutex_要存在？
   → display()可能被采集回调（采集线程上下文）和处理回调（处理线程上下文）同时调用，需要互斥
2. 组件析构顺序为什么重要？
   → 必须先停采集线程（它引用device），再停处理线程，最后关设备。反过来的话线程可能操作已关闭的设备
3. 为什么拍照不能直接调dequeue_buffer？
   → 采集线程在持续DQBUF，主线程再DQBUF就是两个线程并发操作同一个设备的ioctl，会竞争。改为从回调缓存取帧
4. cached_frame_为什么要clone？
   → 采集线程的frame指向的是V4L2 mmap缓冲区，下一帧到来后内容会被覆盖。clone保证缓存的帧数据独立

---

### 第7步：main.cpp（最后看）

只有30多行，但信号处理值得理解：

> "信号处理器里不能做复杂操作——POSIX只允许调用async-signal-safe的函数。我的`request_stop()`里只做了`running_.store(false)`，一次原子写，满足要求。printf严格来说不是signal-safe的，但在实际嵌入式Linux上问题不大。
>
> 关键链路：SIGINT → signal_handler → request_stop() → running_=false → getline返回后检查标志退出 → 析构函数清理所有资源。"
