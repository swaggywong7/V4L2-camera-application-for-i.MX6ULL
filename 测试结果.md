**失败1: `SaveMat_EmptyMat_ReturnsEmptyPath`**

```
失败原因：
  cv::imwrite() 在收到空 Mat 时抛出 cv::Exception（断言失败），
  而非返回 false。
  photo_manager.cpp 的 save_mat() 未对空图像提前校验，
  导致异常传播至调用方。

现象：
  OpenCV(4.6.0): Assertion failed (!_img.empty()) in imwrite

+     if (image.empty()) return "";   // ← 增加此行
      std::string filename = ...
      if (!cv::imwrite(filename, image)) { ... }
```

**失败2: `ConcurrentSave_NoCorruption`**

```
失败原因：
  4个线程同时调用 save_mat()，文件名生成依赖毫秒级时间戳，
  在高并发场景下多个线程可能在同一毫秒内执行，
  导致文件名碰撞（后写覆盖先写）。

现象（日志）：
  photo_20260223_212514_510.jpg  ← 出现两次
  photo_20260223_212514_524.jpg  ← 出现两次

+     static std::atomic<int> seq{0};
```

### 4.5 根因分析

测试揭示了与预期不同的结论：**CPU 被打满的主因不是 OpenCV 算法，而是 MJPEG 解码。**

- mode 0（不调 OpenCV）与 mode 6（调最重的 bilateralFilter）CPU 相差仅 **0.8%**
- 所有 mode 的帧率都在 8–10fps，说明每帧处理耗时约 100ms，
  其中 `cv::imdecode` 是主导项
- mode 0 的 sys% 高达 18.4%，说明 framebuffer 写入是第二大开销

**优化优先级（基于本次测量结果重新排序）**：

1. **降低解码开销（最高优先级）**：将摄像头配置为 YUYV 格式，
   省去软件 JPEG 解码，直接做 YUV→BGR 色彩转换，预计可将帧率从 10fps 提升至 25fps+
2. **降低分辨率**：从 640×480 降至 320×240，MJPEG 数据量缩减约 4 倍
3. **减少 framebuffer 写入**：当场景静止时跳过写入（节省 ~18% sys CPU）
4. **OpenCV 算法优化**（已做的 clone 消除、bilateralFilter 参数调小等）
   在当前瓶颈下帧率提升有限，但对降低 CPU 峰值仍有边际价值


为什么用 top 而不用 perf？

你的板子 PMU 没注册，perf 的硬件计数器（cycles、cache-miss）用不了。perf 虽然有软件事件模式（-e cpu-clock）仍然可用，但对于我们要测的三个指标——进程 CPU 占用率、物理内存、帧率——top + /proc 完全够用，引入 perf 反而增加复杂度。

为什么采样 2 秒？

1 秒太短，top 的 CPU% 是两次快照的差值，1 秒窗口对单核 ARM 来说噪声较大；5 秒太长，模式切换后有几秒的"热身期"数据会被混入。2 秒是常见的经验值，60 秒内可以拿到 30 个样本，足够平均掉抖动。

mode 切换是否 flush buffer？

不会显式 flush。submit_frame() 里只有一个单槽 pending_frame_，新帧直接覆盖旧帧。mode 切换是写 atomic<ProcessMode>，处理线程下一次取帧时读到新 mode。最差情况是最后一帧用旧 mode 处理，下一帧才生效——这个 race window 最多一帧（约 100ms），对性能测量无影响。

内存上涨是否可能是缓存而非泄漏？

完全可能，而且本次数据就是这种情况。VmRSS 包含：进程实际用的堆 + OpenCV buffer pool + OS 分配给进程但尚未回收的页面。我们看到的 3.8MB 波动，是 OpenCV 内部的 Mat 内存池在每帧处理后释放再分配造成的振荡，不是泄漏。

你怎么验证真正没有泄漏？

判断标准是曲线形状而非绝对值：泄漏的特征是内存随帧数线性单调递增，不收敛；我们的数据是在 11MB 附近振荡 3 分钟，首尾相近，没有上升趋势，可判定无泄漏。

更严格的方法：

AddressSanitizer：用 -fsanitize=address 重新编译，在宿主机 QEMU 上运行，ASan 能精确报告每一次越界和泄漏的分配点，比 Valgrind 轻很多，ARM 交叉编译也支持
更长时间采样：如果要面试时说得更有底气，可以把稳定性测试从 3 分钟延长到 30 分钟，处理帧数从几千帧增加到几万帧，趋势更显著

### 6.2 优化一：MJPEG → YUYV 格式切换

**问题**：初始版本用 MJPEG，驱动每帧给出 JPEG 压缩数据，需要调用 `cv::imdecode` 在 CPU 做软件解码。在 528MHz Cortex-A7 上解码一帧 640×480 JPEG 约需 100ms，直接决定帧率天花板约为 10fps。

**关键发现**：Test 1 中 mode 0 和 mode 6 的 CPU 差仅 0.8%（87.7% vs 88.5%），**不是因为两种算法开销相似，而是 CPU 已经满载——MJPEG decode 把单核打满了，bilateralFilter 根本没有 headroom 可以体现出来**。这是"先测量再优化"的典型案例。

**方案**：优先协商 YUYV 格式，失败则降级 MJPEG：

```cpp
int ret = device_->open(w, h, V4L2_PIX_FMT_YUYV);
if (ret < 0) {
    ret = device_->open(w, h, V4L2_PIX_FMT_MJPEG);  // 兼容性回退
}
```

YUYV 是 YUV 4:2:2 原始格式，驱动直接给出未压缩数据，只需做色彩空间转换（YUV→BGR），计算量远小于 JPEG 解码。

**效果**：帧率 9.9 → 13.9fps（+40%），mode 0 CPU 87.7% → 78.6%（−9.1pp）。

---

### 6.3 优化二：OpenCV 算法参数与内存分配优化

**背景**：格式切换后 CPU 从 88% 降至 78%，瓶颈已从 decode 转移到色彩转换和 OpenCV 算法。针对算法层做以下优化：

**① bilateralFilter 参数调小**

| 参数 | 优化前 | 优化后 | 原理 |
|------|--------|--------|------|
| 核大小 d | 9 | **5** | 复杂度 O(d²)，d 减半约降低 4 倍计算量 |
| sigmaColor / sigmaSpace | 300 | **75** | 影响权重范围，适当缩小不明显影响画质 |
| medianBlur 核 | 7 | **5** | O(r²) 复杂度同理 |

**② 去除 Canny 前的冗余 GaussianBlur**

Canny 算法内部第一步就是 Gaussian 平滑，原代码在调用 `cv::Canny` 前额外做了一次 `cv::GaussianBlur`，是纯粹的重复计算，直接删除。

**③ 消除逐帧内存分配**

原代码中各算法的中间 `cv::Mat`（gray、edges、output）每帧都在函数内局部创建，导致每帧都有 `malloc/free`。改为**成员变量复用**：第一帧分配内存，后续帧直接覆盖写入，消除 heap 分配开销。

```cpp
// 优化前：每帧在栈上局部创建，函数退出时释放
cv::Mat process(const cv::Mat& input) {
    cv::Mat gray, edges, output;   // 每帧 malloc
    ...
}
// 优化后：成员变量，跨帧复用已分配的内存
class EdgeDetectAlgorithm {
    cv::Mat gray_, edges_, output_;  // 仅首帧分配
};
```

**④ SharpenKernel / EmbossKernel 构造期初始化**

卷积核是常量，原代码每帧重建，改为在构造函数中初始化一次。

**⑤ submit_frame() 零拷贝**

```cpp
// 优化前：clone() 深拷贝整帧数据（640×480×3 = 921KB/帧）
pending_frame_ = frame.clone();
// 优化后：cv::Mat 引用计数赋值，O(1) 操作，零内存拷贝
pending_frame_ = frame;
```

**效果**：mode 6 单帧时延 113ms → 64ms（−43%）。

---

### 6.4 优化三：静止帧检测，跳过无效 framebuffer 写入

**问题**：每帧向 `/dev/fb` 写入约 1.2MB（640×480×4 字节 ARGB），对系统调用是纯 memcpy 进内核，当画面静止时完全是无效写入。Test 2 中 mode 0 的 sys%（内核态）高达 28.5%，主要来源就是这里。

**方案**：对每帧内容做**滚动哈希**（采样每隔 8 行的中央像素，LCG 混合），哈希相同则跳过写入：

```cpp
uint64_t h = 0;
for (int y = 0; y < frame.rows; y += 8) {
    const uchar* p = frame.ptr(y) + (frame.cols / 2) * 3;
    h = h * 6364136223846793005ULL
        ^ (uint64_t(p[0]) | (uint64_t(p[1])<<8) | (uint64_t(p[2])<<16));
}
if (h == last_frame_hash_) return;  // 内容未变，跳过
last_frame_hash_ = h;
```

哈希计算只读 60 个像素（480/8 行），开销可忽略；碰撞概率约 1/2⁶⁴，不会误跳帧。

---

### 6.5 优化四：ARM NEON 手写 YUV→BGR 色彩转换

**问题**：Test 2 中 mode 0 的 usr%（用户态）高达 61.1%，主要来自 `cv::cvtColor(YUV2BGR_YUYV)`。精简静态编译的 OpenCV **未开启 NEON 路径**，走纯标量代码——640×480 像素每帧约需 10~20 cycle/px，在 528MHz 下占用约 60% CPU。

**YUYV 内存格局**（每 4 字节 = 2 像素）：
```
[Y0][U][Y1][V] [Y2][U][Y3][V] ...
```

**BT.601 整数转换公式**（用 int32 避免溢出，298×255=75990 > INT16\_MAX）：
```
C = Y−16,  D = U−128,  E = V−128
R = (298C + 409E + 128) >> 8
G = (298C − 100D − 208E + 128) >> 8
B = (298C + 516D + 128) >> 8
```

**NEON 实现策略**（每次处理 16 像素 = 32 字节 YUYV → 48 字节 BGR）：

```
vld4_u8(32字节)   → val[0]=Y偶×8, val[1]=U×8, val[2]=Y奇×8, val[3]=V×8
                    （硬件自动解交织，一条指令完成）
vmovl_u8          → uint8×8 扩展到 int16×8（为有符号运算准备）
vsubq_s16         → C=Y−16, D=U−128, E=V−128
vmull_s16         → 298×C，int32×4（防止溢出）
vmlal_s16         → +409E（R），+516D（B）
vmlsl_s16         → −100D, −208E（G）
vshrn_n_s32(8)   → >>8，int32→int16
vqmovun_s16       → 饱和截断到 uint8，自动 clamp [0,255]
vzip_u8           → 将偶/奇像素重新交织，恢复原始像素顺序
vst3_u8           → BGR 交织写出内存
```

`#ifdef __ARM_NEON` 保护，非 ARM 宿主机自动回退 `cv::cvtColor`，尾部不足 16 像素走标量回退。

**效果**（Test 2 → Test 3，全速 framebuffer 条件下）：
- mode 0：78.6% → **17.7%**（−60.9pp，降幅 77%）
- mode 6：90.0% → **22.3%**（−67.7pp）
- bilateralFilter 真实开销首次可量化：22.3 − 17.7 = **4.6pp**（之前因 CPU 满载无法区分）

---

### 6.6 优化五：低帧率预览（500ms 门控）+ 拍照立即显示

**问题**：Test 3 中 mode 0 仍有 17.7% CPU，其中 sys% 约 18% 来自全速 framebuffer 写入（14fps × 1.2MB/帧 = 16.8MB/s 进内核）。

**产品分析**：嵌入式相机的核心诉求是**拍照**，实时预览只是辅助功能。将预览帧率从 14fps 降至 2fps 不影响核心体验，但可减少约 15 倍的 fb 写入次数。

**方案**：`on_frame_processed()` 加时间门控，每 500ms 最多写一次 framebuffer：

```cpp
auto now = std::chrono::steady_clock::now();
if (chrono::duration_cast<chrono::milliseconds>(
        now - last_preview_ts_).count() < kPreviewIntervalMs)
    return;  // 未到 500ms，跳过
last_preview_ts_ = now;
display_->display(frame);
```

**拍照时绕过门控，立即显示**——类似数码相机按快门后屏幕立刻呈现照片：

```cpp
// handle_capture() 中：
display_->display(frame);          // 立即显示，不等 500ms
last_frame_hash_ = 0;              // 强制下次预览刷新
last_preview_ts_ = Clock::now();   // 重置计时器
```

**效果**（Test 3 → Test 4）：mode 0 CPU 17.7% → **8.5%**（−9.2pp）。
