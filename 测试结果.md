#### ⚠️ 失败项分析

**失败1: `SaveMat_EmptyMat_ReturnsEmptyPath`**

```
失败原因：
  cv::imwrite() 在收到空 Mat 时抛出 cv::Exception（断言失败），
  而非返回 false。
  photo_manager.cpp 的 save_mat() 未对空图像提前校验，
  导致异常传播至调用方。

现象：
  OpenCV(4.6.0): Assertion failed (!_img.empty()) in imwrite

严重程度：中等
影响范围：若业务代码向 save_mat() 传入空帧，程序将抛出未捕获异常

建议修复（photo_manager.cpp:62）：
  std::string PhotoManager::save_mat(const cv::Mat& image) {
+     if (image.empty()) return "";   // ← 增加此行
      std::string filename = ...
      if (!cv::imwrite(filename, image)) { ... }
```

---

**失败2: `ConcurrentSave_NoCorruption`**

```
失败原因：
  4个线程同时调用 save_mat()，文件名生成依赖毫秒级时间戳，
  在高并发场景下多个线程可能在同一毫秒内执行，
  导致文件名碰撞（后写覆盖先写）。

现象（日志）：
  photo_20260223_212514_510.jpg  ← 出现两次
  photo_20260223_212514_524.jpg  ← 出现两次
  photo_20260223_212514_535.jpg  ← 出现两次
  预期保存 20 张，实际保存 17 张（3次碰撞）

严重程度：中等
影响范围：高频拍照（同一毫秒内连续拍照）时会丢失照片

建议修复（photo_manager.cpp:31）：
  std::string PhotoManager::generate_filename() const {
      ...
+     // 加入原子计数器避免同毫秒碰撞
+     static std::atomic<int> seq{0};
+     int n = seq.fetch_add(1, std::memory_order_relaxed);
      return std::string(buf) + "_" + std::to_string(ms)
+                             + "_" + std::to_string(n);
  }
```

## 四、真实硬件性能测试（i.MX6ULL 开发板）

> **测试日期**：2026-02-25
> **平台**：i.MX6ULL (ARM Cortex-A7 @ 528MHz, 512MB RAM)
> **摄像头**：USB 摄像头 /dev/video1，640×480 MJPEG
> **测试工具**：BusyBox top (v1.31.1) + 代码内 FPS 计数器 + /proc/[pid]/status
> **采样方式**：每个 mode 采集 60 秒（30次 × 2s），内存稳定性测试持续 3 分钟

---

### 4.1 帧率（FPS）

| 测试场景 | 平均 FPS | 样本数 |
|----------|----------|--------|
| mode 0 — 不处理（基线） | **9.9 fps** | 58 |
| mode 1 — 高斯模糊 | **7.8 fps** | 58 |
| mode 2 — Canny 边缘检测 | **7.7 fps** | 58 |
| mode 6 — 卡通化 | **7.8 fps** | 58 |

**发现**：全部 mode 的帧率集中在 8–10 fps，远低于摄像头硬件上限 30fps。
各 mode 间差异仅约 2fps，说明瓶颈不在 OpenCV 算法，而在更前面的
**MJPEG 解码**（`cv::imdecode`）。在 528MHz Cortex-A7 上解码一帧 640×480 MJPEG 约需 100ms，
决定了整条处理链的帧率天花板约为 10fps。

---

### 4.2 CPU 占用

#### 进程 CPU 占用率

| 测试场景 | 平均 %CPU | 最高 %CPU |
|----------|-----------|-----------|
| mode 0 — 基线 | 87.7% | 93% |
| mode 1 — 高斯模糊 | 88.1% | 93% |
| mode 2 — Canny | 87.4% | 93% |
| mode 6 — 卡通化 | 88.5% | 93% |

#### 用户态 / 内核态分解

| 测试场景 | usr%（用户态） | sys%（内核态） | 解读 |
|----------|----------------|----------------|------|
| mode 0 — 基线 | 80.2% | 18.4% | sys 高：framebuffer memcpy 进内核代价大 |
| mode 1 — 高斯 | 90.1% | 8.5% | usr 上升：OpenCV 计算密集 |
| mode 2 — Canny | 83.8% | 14.9% | 中间值 |
| mode 6 — 卡通化 | 91.4% | 7.3% | usr 最高：bilateralFilter 极度计算密集 |

**发现**：CPU 在所有 mode 下均维持在 **87–89%**，峰值 93%，系统接近满载，无扩展余量。
mode 0 的 sys% 高达 18.4%，来自每帧向 /dev/fb 写入约 1.2MB（640×480×4字节）的内核态 memcpy。
这意味着即使不做任何图像处理，仅"显示"本身就占用约 18% 的 CPU。

### 4.5 根因分析

测试揭示了与预期不同的结论：**CPU 被打满的主因不是 OpenCV 算法，而是 MJPEG 解码。**

- mode 0（不调 OpenCV）与 mode 6（调最重的 bilateralFilter）CPU 相差仅 **0.8%**
- 所有 mode 的帧率都在 8–10fps，说明每帧处理耗时约 100ms，
  其中 `cv::imdecode` 是主导项
- mode 0 的 sys% 高达 18.4%，说明 framebuffer 写入是第二大开销

**优化优先级（基于本次测量结果重新排序）**：

1. **降低解码开销（最高优先级）**：将摄像头配置为 YUYV 格式，
   省去软件 JPEG 解码，直接做 YUV→BGR 色彩转换，预计可将帧率从 10fps 提升至 25fps+
2. **降低分辨率**：从 640×480 降至 320×240，MJPEG 数据量缩减约 4 倍
3. **减少 framebuffer 写入**：当场景静止时跳过写入（节省 ~18% sys CPU）
4. **OpenCV 算法优化**（已做的 clone 消除、bilateralFilter 参数调小等）
   在当前瓶颈下帧率提升有限，但对降低 CPU 峰值仍有边际价值
