# 性能测试方案：三项指标量化分析

> 针对"你是怎么验证这个项目性能的？""你怎么知道性能够不够用？"这类问题

---

## 一、测试背景与动机

i.MX6ULL 是 ARM Cortex-A7 单核处理器，主频 528MHz，内存最大 512MB。
这类嵌入式平台不像 PC 有充足的计算冗余——如果图像处理太慢，
采集帧就会堆积，最终表现为画面卡顿或内存不断增长。

所以在上板子之前，我有三个需要量化回答的问题：

1. **帧率能不能达到实时？** 30fps 是摄像头输出上限，处理链跟不上就会丢帧。
2. **CPU 有没有被打满？** 单核平台如果 CPU 占用过高，系统会失去响应余量，
   也没有资源留给未来扩展（比如加网络发送、加语音采集）。
3. **内存有没有泄漏？** 嵌入式设备可能要连续运行数小时，
   哪怕每帧泄漏 1KB，运行一小时就会泄漏掉 100MB+，最终 OOM 崩溃。

这三个指标分别对应：**吞吐能力、计算负载、资源健康度**，缺一不可。

---

## 二、指标一：帧率（FPS）

### 测试目的

验证在不同处理模式下，采集+处理的完整链路能否维持足够的帧率。
帧率下降是处理线程成为瓶颈的直接体现。

### 测什么数据

- **采集帧率**：`capture_loop()` 实际每秒拿到多少帧（反映 V4L2 驱动端）
- **处理帧率**：`ImageProcessor` 每秒完成多少帧的处理（反映 OpenCV 算法端）
- **两者差值**：差值大说明帧覆盖机制在工作——处理跟不上，旧帧被新帧覆盖丢弃

### 测试方法

在 `capture_loop()` 里加计数器，每秒输出一次 FPS：

```cpp
auto fps_start = std::chrono::steady_clock::now();
int fps_count = 0;
// 每次拿到有效帧后：
fps_count++;
auto elapsed_ms = std::chrono::duration_cast<std::chrono::milliseconds>(
    std::chrono::steady_clock::now() - fps_start).count();
if (elapsed_ms >= 1000) {
    printf("[PERF] FPS: %.1f\n", fps_count * 1000.0f / elapsed_ms);
    fps_count = 0;
    fps_start = std::chrono::steady_clock::now();
}
```

### 怎么看结果

| 现象 | 结论 |
|------|------|
| 各 mode 下 FPS 接近（≥25fps） | 处理线程不是瓶颈，系统有余量 |
| 复杂 mode（卡通化）FPS 明显低 | 该算法成为瓶颈，需要优化 |
| FPS 随时间下降 | 内存或资源异常，需排查 |

## 三、指标二：CPU 占用

### 测试目的

量化不同处理模式对 CPU 的实际消耗，确认：
1. 最重的处理模式下 CPU 占用不超过 80%（留系统余量）
2. 用户态（us）和内核态（sy）的比例是否合理

### 测什么数据

- **%CPU**：进程总 CPU 占用率
- **us vs sy 的比例**：us 高说明瓶颈在用户代码（OpenCV 处理），
  sy 高说明瓶颈在内核（V4L2 驱动的 ioctl、framebuffer 写入）

### 测试方法

用 `top` 批处理模式，对运行中的进程定时采样：

```bash
# 程序启动后，找到 PID
PID=$(pidof v4l2_camera)

# 采样 60 次，每 2 秒一次，输出到文件
top -b -n 60 -d 2 -p $PID > /tmp/cpu_log.txt
```

`top` 的数据来自 `/proc/[pid]/stat`，是内核调度器的精确计数，
和硬件 PMU 无关，在嵌入式板子上同样可靠。

### 怎么看结果

关注两个层面：

**进程列表行**（v4l2_camera 那一行的 `%CPU` 列）
```
  PID   %CPU  %MEM   COMMAND
 1234   45.3   2.4   v4l2_camera
```

**CPU 汇总行**（区分用户态和内核态）
```
%Cpu(s): 38.2 us, 7.1 sy, 0.0 ni, 54.2 id
          ↑          ↑
       OpenCV处理  V4L2驱动 ioctl / framebuffer 写
```

| 观测点 | 判断标准 |
|--------|----------|
| mode 0 时 %CPU | 基线，主要是 memcpy 到 framebuffer 的开销 |
| mode 1-6 时 %CPU 增量 | 每种算法的额外 CPU 代价 |
| sy 占比是否异常高 | >20% 需要检查 ioctl 频率或 framebuffer 写入效率 |

## 四、指标三：内存占用

### 测试目的

验证长时间运行下内存没有持续增长——即没有内存泄漏。
同时确认峰值内存在板子可接受范围内。

### 测什么数据

- **RES（物理内存）**：top 里的 RES 列，对应 `/proc/[pid]/status` 的 `VmRSS`，
  这是进程**实际占用**的物理内存，是真正有意义的数字
- **VIRT（虚拟内存）**：不代表实际内存消耗，mmap 的未访问区域也算在内，
  这个数字偏大是正常的，不需要紧张
- **内存随时间的变化趋势**：泄漏的特征是线性增长；正常的初始化开销是增长后趋于平稳

### 测试方法

**方法一：从 top 输出里提取 RES 列**

```bash
# 采样 120 次（4 分钟），提取时间和 RES 列
top -b -n 120 -d 2 -p $PID | grep v4l2_camera | awk '{print NR, $6}' > /tmp/mem_log.txt
```

**方法二：直接读 /proc（更精确）**

```bash
while [ -d /proc/$PID ]; do
    grep VmRSS /proc/$PID/status | awk -v t="$(date +%H:%M:%S)" '{print t, $2, "kB"}'
    sleep 5
done > /tmp/mem_trend.txt
```

### 怎么看结果

| 现象 | 结论 |
|------|------|
| 前 30 秒轻微增长，之后稳定 | 正常（OpenCV 懒加载初始化，分配一次后复用） |
| 内存随帧数线性递增，不收敛 | 存在泄漏，检查 cv::Mat clone 后是否正确释放 |
| 拍照时出现内存尖峰 | 正常，imwrite 编码时临时分配，用完即释放 |

## 五、测试场景设计

针对 7 种处理 mode，设计对比测试：

| 测试场景 | 持续时间 | 目的 |
|----------|----------|------|
| mode 0（不处理，基线） | 2 分钟 | 建立 CPU/内存/FPS 基准值 |
| mode 1（高斯模糊） | 2 分钟 | 滤波算法的额外开销 |
| mode 2（Canny 边缘检测） | 2 分钟 | 边缘检测算法的额外开销 |
| mode 6（卡通化） | 2 分钟 | 最复杂算法，预期开销最大 |
| mode 6（卡通化，长跑） | 30 分钟 | 长时间稳定性，主要看内存趋势 |

## 六、面试官常问 Q&A

### Q: "你为什么选这三个指标？"

> 这三个指标覆盖了嵌入式实时系统最核心的三个维度：
> - FPS 衡量的是**时效性**——结果来得够不够快
> - CPU 衡量的是**计算负载**——系统有没有被打满，有没有余量
> - 内存衡量的是**长期健康度**——能不能跑 24 小时不崩
>
> 缺任何一个都不完整：FPS 高但 CPU 100% 说明没有扩展余量；
> FPS 和 CPU 都正常但内存泄漏，设备迟早 OOM。

---

### Q: "为什么用 top 而不是 perf？"

> 开发板的内核 PMU 没有注册，perf 依赖硬件性能计数器的功能用不了。
> 但 top 的数据来源是 /proc 文件系统，是内核调度器的直接输出，不依赖 PMU，
> 完全正常。
>
> 对于我的三个指标来说 top 完全够用：%CPU 和内存占用直接可见，
> FPS 需要在代码里打印，top 管不到这个，所以加了代码级计数器。
>
> perf 的优势是能看到**函数级**的热点，知道 CPU 时间花在哪个函数上。
> 如果 top 发现 CPU 占用高，下一步才需要 perf 来定位是哪个函数的锅。

---

### Q: "你怎么区分内存泄漏和正常的内存增长？"

> 判断依据是增长的**曲线形状**而不是绝对值。
>
> 正常的初始化开销：增长发生在程序启动的前几十秒（OpenCV 第一次调用某个函数时
> 会懒加载内部资源），之后内存趋于平稳，不随帧数增长。
>
> 真正的泄漏：内存随帧数或运行时间单调递增，没有收敛趋势。
> 如果处理了 10000 帧后内存比 1000 帧时明显高，且差值和帧数大致成比例，
> 就是泄漏。
>
> 这个方法是嵌入式上 Valgrind 用不了时的替代手段，零依赖，只读 /proc。
