## 二、测试体系设计（面试官追问：你的测试怎么组织的）

### 为什么不用 gtest？

> gtest 交叉编译到 ARM 需要单独配置 CMake 工具链，还要解决 C++ ABI 兼容性问题，
> 依赖配置本身就很复杂。我的项目只需要基础的断言和测试注册功能，
> 所以自己写了一个不到 120 行的框架，核心是两个东西：
>
> 1. `TEST(模块, 用例)` 宏 —— 用静态变量的构造时机，在 main 执行前就把测试函数注册到全局列表里
> 2. `run_all_tests()` —— 遍历列表执行，每个测试用 try-catch 隔离，一个失败不影响后续
>
> 这样新增测试只需要写一个 `TEST(...)` 块，不需要修改任何注册代码。
> 编译出来的 ARM 二进制可以直接 scp 到开发板上跑。

### 测试分层

```
硬件集成测试（手工）        ← 需要 /dev/video0、/dev/fb0，真实设备才能跑
      ↑
压力测试 stress_test        ← 6项，模拟长时间运行、内存泄漏、多线程并发
      ↑
单元测试 unit_tests         ← 47项，不依赖任何硬件，CI/主机/开发板均可运行
```

**为什么能在 x86 主机上跑 ARM 的单元测试？**
> 我把单元测试设计成只依赖 OpenCV 和标准库，不涉及 V4L2 设备和 FrameBuffer，
> 所以可以在主机上用 qemu-arm-static 仿真运行，在开发板上也能原生运行，
> 不需要每次改完代码都烧板子验证。

## 三、测试发现的两个 Bug

### BUG-001：传空图像会崩溃

**怎么发现的**

> 我在测试 `save_mat()` 的边界条件时，写了一个用例：传入空 `cv::Mat`，预期返回空字符串。
> 实际跑起来程序直接崩了，不是返回错误，是抛了异常。

**根因**

> `cv::imwrite` 遇到空 Mat 会触发内部断言 `!_img.empty()`，抛出 `cv::Exception`。
> 但 `save_mat()` 没有捕获它，异常直接往上传播。
> 在实际使用中，如果采集线程刚启动还没拿到第一帧，用户就按了拍照键，
> 传进来的就是空 Mat，程序就崩了。

**修复**

```cpp
// 修复前：直接调 imwrite，空图会抛异常
std::string PhotoManager::save_mat(const cv::Mat& image) {
    std::string filename = ...
    if (!cv::imwrite(filename, image)) { ... }
}

// 修复后：入口提前校验
std::string PhotoManager::save_mat(const cv::Mat& image) {
    if (image.empty()) return "";        // ← 加这一行
    std::string filename = ...
    if (!cv::imwrite(filename, image)) { ... }
}
```

### BUG-002：多线程并发保存照片会丢文件

**怎么发现的**

> 压力测试里有一个多线程并发保存的用例：4个线程各保存5张，预期20个文件。
> 跑完一数只有17个，少了3张。

**根因**

> 文件名用的是时间戳，精度是毫秒级。
> 4个线程并发执行时，多个线程可能在同一毫秒内拿到完全相同的文件名，
> 后写的直接覆盖了先写的，照片就丢了。

**修复**

```cpp
// 修复前：纯时间戳，并发下会碰撞
return std::string(buf) + "_" + std::to_string(ms);

// 修复后：追加原子序列号，全局唯一
static std::atomic<int> seq{0};
int n = seq.fetch_add(1, std::memory_order_relaxed);
return std::string(buf) + "_" + std::to_string(ms) + "_" + std::to_string(n);
```

> 用 `atomic` 而不是 mutex，是因为这里只需要保证递增操作的原子性，
> 不需要保护一段临界区，`fetch_add` 是最轻量的做法。

## 四、性能测试结果与分析

> **说明：** 以下数据在 qemu-arm 仿真下采集，真实 i.MX6ULL 硬件会更快。

### 吞吐量（高斯模糊，320×240）

| 场景 | 提交帧数 | 实际处理帧数 | 有效 FPS |
|------|---------|------------|---------|
| 极限压测（无 sleep） | 915 帧/10s | 155 帧/10s | 15.5 fps |
| 模拟 30fps 输入 | 216 帧/10s | — | — |

> 提交比处理快很多，说明帧覆盖机制在正常工作——处理不过来的旧帧被自动丢弃，
> 显示的永远是最新帧，这是实时视频的正确策略。

### 内存稳定性（10秒 EdgeDetect 持续运行）

| 指标 | 数值 |
|------|------|
| 起始 RSS | 47,628 KB |
| 结束 RSS | 48,148 KB |
| 内存增长 | **+520 KB** |

> 520 KB 的增长来自 OpenCV 内部的懒加载初始化，不是泄漏。
> 验证方法是观察增长曲线：如果是泄漏，内存会随帧数线性增长；
> 这里是运行了一会儿之后就稳定了，说明没有泄漏。

### 多线程并发（4生产者）

| 指标 | 数值 |
|------|------|
| 总提交帧数 | 3936 帧 |
| 实际处理帧数 | 3124 帧 |
| 竞态错误 | **0** |

## 五、面试官常问 Q&A

### Q: "你的测试覆盖率怎么样？"

> 我没有做代码行覆盖率的统计工具，但我按功能维度覆盖了关键路径：
> - **正常路径**：保存、加载、算法处理的基本功能
> - **边界条件**：空图像、索引越界、空目录
> - **并发场景**：多线程提交帧、多线程并发保存、运行中切换模式
> - **生命周期**：反复启停、重复调用 start/stop、不 start 直接 stop
>
> 实践下来边界条件和并发场景发现问题的概率最高，两个 Bug 都是在这两类用例里暴露的。

---

### Q: "压力测试和单元测试的区别是什么？"

> 单元测试验证"功能对不对"——给一个输入，检查输出是否符合预期，运行时间秒级。
> 压力测试验证"长时间跑会不会垮"——不关心单次结果，关注内存增长趋势、崩溃率、
> 在高负载下的竞态问题，运行时间分钟级。
>
> 两者互补：单元测试快速定位功能 Bug，压力测试暴露只在特定时序下才触发的问题。
> BUG-002（文件名碰撞）在单线程单元测试里跑不出来，只有并发压力测试才能稳定复现。

---

### Q: "你怎么监控内存泄漏的？"

> 在嵌入式 Linux 上没有 Valgrind 这类工具（交叉编译很麻烦），
> 我用的是最直接的方式：在压力测试里每隔5秒读一次 `/proc/self/status` 里的 `VmRSS`，
> 记录内存随时间的变化曲线。
>
> 判断依据：泄漏的内存会随运行时间或帧数单调递增；
> 正常的内存波动会在一个区间内稳定。
> 这个方法不精确，但对于"有没有明显泄漏"的判断足够用，而且零依赖。

---

### Q: "发现 Bug 之后你怎么确认修复正确？"

> 两步：
> 1. 把原来失败的测试用例跑通（BUG-001 和 BUG-002 对应的测试从 FAIL 变 PASS）
> 2. 把其他所有测试也跑一遍，确认修复没有引入新问题
>
> 修复前 47 个测试通过 45 个，修复后 47/47 全部通过。
> 用 git 保存了修复前的版本（初始提交），修复后单独提交，commit message 里
> 写清楚了每个 Bug 的根因和修复方式，方便回溯。

---

### Q: "如果让你继续优化，你会从哪里入手？"

> 根据测试数据，瓶颈在图像处理线程，不在 I/O。所以优化方向有三个：
>
> 1. **算法级**：卡通化用了双边滤波，是最慢的算法，可以降采样后处理再放大，
>    以轻微质量损失换 4x 以上的速度提升。
>
> 2. **帧级**：`submit_frame()` 里有一次 `cv::Mat::clone()`，
>    可以改成双缓冲——两块固定内存交替使用，避免每帧都 malloc/free。
>
> 3. **架构级**：目前是单消费者，可以加一个处理帧队列（深度2），
>    让采集和处理真正流水线化，减少互相等待的时间。
>
> 但在做这些优化之前，我会先在真实开发板上跑一次性能测试，
> 确认瓶颈确实是处理线程，再动手——不猜瓶颈，先测量。
